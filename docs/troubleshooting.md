# ハマりどころ記録

遭遇したエラー・ハマりどころ・回避策を時系列で記録する。本の「ハマりどころコラム」の一次素材。エラーメッセージは原文のまま残すこと。

## 2026-07-14: `.zshrc` への追記が既存行と連結して壊れた

環境変数の永続化のため `.zshrc` に `source $HOME/export-esp.sh` を追記した際、既存の最終行に改行が無かったため、追記内容が前の行と連結されて `.zshrc` が壊れた。

- **症状**: 新しいシェルで環境変数が読み込まれない(壊れた行がエラーになる)
- **回避策**: `.zshrc` を開いて手動で2行に分離
- **教訓**: `echo >> ~/.zshrc` で追記する前に、ファイル末尾に改行があるか確認する(`tail -c 1 ~/.zshrc | xxd` で確認できる)

## 2026-07-19: `cargo run` に espflash のオプションを渡すには `--` が必要

シリアルポートを明示指定しようと `cargo run --port /dev/cu.usbmodem1101` と実行したら cargo に拒否された。

```
error: unexpected argument '--port' found

  tip: to pass '--port' as a value, use '-- --port'
```

- **原因**: `--port` は cargo のオプションではなく、runner(espflash)のオプション。cargo に渡す引数と runner に渡す引数は `--` で区切る
- **回避策**: `cargo run -- --port /dev/cu.usbmodem1101` とする。ポートが1つしか無い場合は espflash が自動検出するので省略も可

## 2026-07-19: esp-idf-hal 0.46 で `prelude` モジュールが廃止されている

ネット上のサンプルコードによくある `use esp_idf_hal::prelude::*;` が 0.46.2(git版)ではコンパイルエラーになる。

```
error[E0432]: unresolved import `esp_idf_hal::prelude`
  --> src/main.rs:11:5
   |
11 |     prelude::*,
   |     ^^^^^^^ could not find `prelude` in `esp_idf_hal`

error[E0689]: can't call method `kHz` on ambiguous numeric type `{integer}`
```

- **原因**: esp-idf-hal の破壊的変更で prelude が削除された。`.kHz()` / `.MHz()` などの単位変換メソッドは prelude 経由で入っていた
- **回避策**: `use esp_idf_hal::units::FromValueType;` を明示的にインポートする
- **教訓**: esp-idf-hal 系は破壊的変更が多い。ネットのサンプルはバージョン確認必須

## 2026-07-20: espflash monitor の出力はパイプ経由だとバッファリングされて届かない

`cargo run ... | grep ...` のようにシリアルモニタの出力をパイプで加工すると、espflash側の標準出力がブロックバッファリング(約8KB単位)になり、アプリのログがいつまでもパイプ先に流れてこない。前回はブートログが途中で切れた状態のまま止まって見えた。

- **症状**: 実機は正常動作しているのに、パイプ先のログファイルが空 or ブートログ途中で止まる
- **原因**: 標準出力がTTYでない場合のlibcのバッファリング挙動
- **回避策**: ログをファイルに落として加工したい場合は `script -q /dev/null cargo run ...` のように疑似TTYを噛ませる。目視デバッグならターミナルで直接 `cargo run` すれば問題ない

## 2026-07-20: Wi-Fi接続が「認証OK→アソシエーションで切断」を繰り返す

親機のSSIDを指定したところ、以下のパターンで接続に失敗し続けた。

```
I (3875) wifi:state: init -> auth (0xb0)
I (3885) wifi:state: auth -> assoc (0x0)
I (4885) wifi:state: assoc -> init (0x400)
Wi-Fi接続に失敗(SSID/パスワードを確認): ESP_ERR_TIMEOUT (error code 263)
```

- **症状**: 電波は見えていて認証(auth)まで通るが、アソシエーション段階で切断されタイムアウト
- **解決**: 接続先を中継器のSSID(`..._EXT`)に変更したら一発で接続できた。台所のデバイス設置場所からは親機が遠く、中継器経由が正解だった
- **教訓その1**: 「パスワード間違い」なら4-wayハンドシェイク段階で失敗する。アソシエーションで落ちる場合は電波状況・ルーター側の事情を疑う
- **教訓その2**: この過程で接続リトライループ(最大5回)を実装した。ESP32のWi-Fi初回接続は一発で通らないことがあるので、リトライは必須の設計

## 2026-07-20: シリアルログのバッファリングは `script` コマンドでも解決せず → 画面表示が最強

espflash monitor の出力を疑似TTY(`script -q /dev/null`)経由でファイルに落としても、やはり途中でバッファに滞留して読めなくなる現象が再発。最終的にWi-Fi接続結果は「LCDにIPアドレスを描画する」ことで確認した。

- **教訓**: 組込み開発では「デバイス自身の画面にステータスを出す」のが最も確実なデバッグ手段。シリアルはあくまで補助と考える(本番の台所運用ではUSBを繋がない前提でもあり、画面表示ファーストの設計は今後も踏襲する)

## (事前調査で判明)SDカードは FAT32 必須

CoreS3 で使う SD カードは FAT32 でフォーマットされている必要がある。64GB 以上のカードは標準で exFAT のため、そのままでは認識しない。16GB(FAT32)を使用する。
