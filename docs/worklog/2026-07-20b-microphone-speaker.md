## やったこと

1. 最難関と見ていたマイク録音区間(ES7210 + I2S受信)→ RMSレベルメーターで検証、**一発で動いた**
2. スピーカー再生(AW88298 + I2S送信)→ タッチでビープ音、これも一発で動いた

## 進め方(ここが書籍的に重要)

記憶やネット記事でレジスタ値を書かず、一次情報を先に確定してから実装した:

1. **M5UnifiedのソースをWebFetchで取得**し、CoreS3向けES7210初期化のレジスタ/値ペア25個+最終クロック確定(0x01=0x14)を抽出
2. **esp-idf-halのI2S APIは手元のgitチェックアウトを直接grep**して `new_std_rx` のシグネチャを確認(docs.rsはバージョン違いの罠があるため)

```sh
# APIシグネチャの確認に使ったコマンド
grep -rn "new_std_rx" ~/.cargo/git/checkouts/esp-idf-hal-*/*/src/i2s/std.rs
```

## 実装の骨子(コミット参照)

- ES7210(I2C addr 0x40)にM5Unified実績値でレジスタ書き込み(初期化順序ごと踏襲)
- I2S: BCLK=GPIO34, WS=GPIO33, DIN=GPIO14, MCLK=GPIO0 / ESP32-S3マスター、16kHz・16bit・ステレオ
- メインループ: 512フレーム(32ms)ずつ読み、左チャネルのRMS→dB換算(-60〜0dB)→320px幅の緑バーで表示
- 差分描画(バー幅が変わったときだけ再描画)でSPI帯域を節約

## スピーカー再生(同日追記)

マイクと同じ進め方(M5Unifiedを一次情報に)で続けてスピーカーも実装:

- AW88298(I2C addr 0x36)を初期化。**レジスタは16bit幅・ビッグエンディアン**で書く点がAXP2101等と違う
- サンプルレート依存のレジスタ0x06は計算式 `(rate+1102)/2205` →レートテーブル参照で `0x14C3`(16kHz)
- I2Sドライバを `new_std_rx` → `new_std_bidir` に変更(DOUT=GPIO13追加)。マイクとスピーカーは同一I2Sバス共有
- タッチ時に1kHzサイン波150msを再生 → 実機で「ピッ」を確認。再生中はマイクが自分の音を拾ってメーターが跳ねる(送受同時動作の証明)

## 結果

- 実機で声・拍手に緑バーが反応することを確認(録音→数値化→表示の縦一本が成立)
- タッチ→ビープ再生→マイクが自音を拾う、まで送受同時動作を確認
- ビルド: インクリメンタル5秒前後 / 新規クレート追加なし

## ハマりどころ

- 書き込み時に「Error while connecting to device / No such file or directory」→ シリアルポート名が `/dev/cu.usbmodem1101` から `/dev/cu.usbmodem101` に変わっていた。USBの挿し直しや再列挙でポート名は変わるので、エラー時はまず `ls /dev/cu.*` で現在のポートを確認する

## 計測値

- 音声フレーム: 512フレーム/32ms周期(16kHz)。メインループはI2S読みでペーシングされる
- Wi-Fi接続所要時間の計測は持ち越し(起動経過msの画面表示をまだ入れていない)

## 教訓

- 「動くコードの実績値(M5Unified)を一次情報として先に確保→そのまま使う」が組込みでは最速で確実
- APIのバージョン差は手元の実ソースで潰す。この2点で最難関想定の区間がノーハマりで通った
